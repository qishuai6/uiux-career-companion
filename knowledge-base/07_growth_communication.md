# 成长、复盘、沟通与述职

## 核心判断

职场表达不是汇报工作量，而是让别人看见你的判断、贡献和可复用能力。

设计师成长的关键不是“做了多少项目”，而是每个项目有没有形成：

- 问题定义能力
- 判断依据
- 协作方法
- 结果验证
- 复盘沉淀

## 述职常见问题

- 超时：说明总结和取舍能力不足。
- PPT 逻辑混乱：评委无法沿着你的思路理解价值。
- 只讲项目数量：数量是苦劳，不是功劳。
- 念 PPT：页面成了稿子，不是提词器。
- 只展示视觉成果：看不出设计如何影响业务或体验。

## 述职结构

推荐结构：

1. 今年面对的核心业务/团队问题。
2. 我负责的关键方向。
3. 代表项目 1：问题、判断、动作、结果。
4. 代表项目 2：问题、判断、动作、结果。
5. 方法沉淀：形成了什么机制、规范、流程、工具。
6. 反思：什么没做好，下一阶段怎么补。
7. 明年计划：围绕业务目标和个人能力目标展开。

述职不是作品集，不要每个项目都讲。选最能代表价值的项目。

## 复盘

复盘不要只问“做得好不好”，要问：

- 当初的目标是什么？
- 目标是否被正确拆解？
- 哪些判断是对的？
- 哪些判断被事实推翻？
- 哪些问题来自能力，哪些来自流程，哪些来自资源？
- 下次如何提前规避？

复盘输出要有行动：

- 调整流程
- 补充指标
- 优化模板
- 沉淀组件
- 建立检查清单
- 改进协作机制

## 沟通

设计沟通不要只表达“我觉得更好”，要表达：

- 我们共同目标是什么。
- 当前方案卡在哪里。
- 我基于什么信息做判断。
- 有哪些可选方案。
- 每个方案的收益和风险是什么。
- 建议优先选哪个，为什么。

和产品沟通：

- 对齐业务目标和用户目标。
- 追问需求来源和验证方式。
- 区分必须做、应该做、可以以后做。

和开发沟通：

- 提前说明交互规则、状态、边界条件。
- 用组件和 Token 降低实现歧义。
- 不要到验收阶段才讨论可行性。

和领导沟通：

- 少讲过程堆料，多讲关键判断。
- 用结果和证据支撑设计价值。
- 主动暴露风险和取舍。

## 周报与日常沉淀

周报不要写流水账。

推荐格式：

- 本周完成了什么关键判断。
- 推动了什么项目进展。
- 遇到什么风险。
- 需要谁协助什么。
- 下周重点是什么。

日常沉淀要服务未来：

- 可放进作品集的项目证据。
- 可复用的组件和模式。
- 可复盘的问题和方法。
- 面试能讲的案例素材。

## 成长方向

设计师不要只卷工具。更应该积累：

- 业务理解
- 研究和问题定义
- 交互复杂度处理
- 视觉系统能力
- 数据和验证意识
- 跨团队沟通
- AI 工具化能力

真正的成长是让自己从“能完成页面”变成“能定义问题、组织方案、证明价值”。

# 作品集与 Case Study

## 核心判断

作品集不是介绍项目，而是证明“我能在真实约束里做设计判断”。项目只是载体，主语必须是设计师本人。

好的作品集要回答：

- 我面对的业务/用户问题是什么？
- 为什么这个问题值得做？
- 我负责了什么，主导了什么，参与了什么？
- 我基于什么信息做判断？
- 我选择了什么方案，放弃了什么方案，为什么？
- 结果如何验证？如果没有数据，如何解释验证不足？

## 常见问题

- 只展示 UI 页面，不展示问题定义和判断过程。
- 项目背景过空，只有行业介绍，没有业务目标。
- 用户画像、体验地图、竞品分析堆太多，但看不出为什么要做。
- 没有交代自己的真实工作范围，让面试官怀疑真实性。
- 作品集项目太多，质量不齐，最弱项目会拉低整体判断。
- 样机和模板痕迹太重，看起来像包装能力，不像设计能力。
- 结果只写“提升体验、提高效率”，没有指标、证据或验证逻辑。

## 优秀结构

推荐每个 case study 按这个顺序组织：

1. 项目一句话定义：什么业务、什么用户、解决什么问题。
2. 项目背景：产品阶段、业务目标、用户场景、约束条件。
3. 核心问题：旧方案卡在哪里，问题如何被发现。
4. 我的角色：负责范围、主导事项、协作边界。
5. 设计目标：把业务目标转成体验目标。
6. 设计判断：为什么优先解决这些点，依据是什么。
7. 方案动作：关键动作，每个动作解决什么问题。
8. 结果验证：数据、反馈、可用性测试、上线表现、复盘。
9. 方法沉淀：这次项目让你形成了什么可复用能力。

## 页面表达

- 一页只讲一件事。
- 每页大标题直接写结论，不写空泛标题。
- 先交代判断，再展示结果图。
- 复杂过程用流程图、对比表、优先级矩阵表达。
- UI 图要服务于论证，不要把作品集变成界面截图合集。

## 设计验证

验证不是固定流程，要根据项目类型选择证据。

局部功能优化可看：

- 漏斗变化
- 点击率、转化率、跳出率
- 任务完成时长
- 错误率、投诉量、客服反馈
- 用户是否更容易发现、理解、使用功能

大版本改版可看：

- 满意度、评分、NPS
- DAU、新增、留存、转化
- 问卷、访谈、用户反馈
- 社交平台和应用商店评价

B 端项目可看：

- 操作步骤是否减少
- 任务完成时间是否下降
- 错误率和返工率是否下降
- 组件复用率、规范覆盖率、一致性提升
- 设计和开发协同效率是否提升

如果没有数据，不要编。可以写：

- “该项目没有完整上线数据，所以我用任务路径、可用性走查和关键用户反馈来验证方向。”
- “这里的验证不足是项目限制，复盘时我会补充埋点、任务成功率和用户反馈闭环。”

## 项目介绍框架

当用户提供一段零散项目经历时，先整理成这个框架：

```markdown
# 项目介绍框架

## 1. 项目一句话定义
这个项目是为【目标用户】在【场景】下解决【核心问题】的【产品/功能/系统】。

## 2. 项目背景
- 业务处于什么阶段
- 当前目标是什么
- 为什么现在要做

## 3. 核心问题
- 用户/业务卡在哪里
- 旧方案为什么不够好
- 这些问题如何被发现

## 4. 我的角色
- 我负责什么
- 我主导什么
- 我参与但没有主导什么

## 5. 设计判断
- 优先级如何判断
- 依据来自业务、用户、数据还是专家走查
- 哪些方案被放弃，为什么

## 6. 方案动作
- 动作 1：解决什么问题
- 动作 2：解决什么问题
- 动作 3：解决什么问题

## 7. 结果与验证
- 有数据写数据
- 没数据写验证逻辑和复盘

## 8. 面试追问
- 为什么这么设计？
- 你怎么证明有效？
- 这个项目里你真正主导了什么？
- 如果重做，你会改哪里？
```

## 诊断优先级

诊断作品集时，优先看：

1. 背景是否清楚。
2. 问题是否成立。
3. 角色是否可信。
4. 判断是否可见。
5. 结果是否有证据。
6. 页面是否服务叙事。

缺结构比缺美观更严重。没有判断比视觉普通更危险。

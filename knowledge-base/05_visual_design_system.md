# UI 视觉与设计系统

## 核心判断

视觉不是装饰，设计系统也不是组件截图。UI 的价值在于清晰、一致、可用、可扩展，并能和开发落地形成闭环。

## UI 诊断

常见“不高级”不是因为缺装饰，而是因为：

- 间距没有秩序。
- 字号层级混乱。
- 对齐关系不稳定。
- 色彩没有语义。
- 组件状态不完整。
- 页面里每个模块都想抢注意力。
- 内容密度和业务场景不匹配。

诊断顺序：

1. 信息层级是否清楚。
2. 阅读动线是否顺。
3. 间距和对齐是否有规则。
4. 色彩是否服务状态和重点。
5. 组件是否一致。
6. 是否有多余装饰影响理解。

## 设计系统

设计系统不是一次性交付物，而是服务产品的产品。

它通常包括：

- 设计原则
- 设计语言
- 设计模式
- 组件库
- Design Token
- 使用指南
- 开发实现
- 治理和迭代机制

只停留在 Figma 组件库，不算完整设计系统。真正有价值的设计系统要连接设计和工程。

## 设计原则

设计原则必须能指导取舍。

弱原则：

- 美观
- 简洁
- 高效
- 一致

强原则：

- 清晰优先于装饰。
- 关键任务优先于视觉完整性。
- 复杂功能优先保障可解释性。
- 同一业务对象在不同页面保持一致表达。

原则要有优先级。否则当“美观”和“效率”冲突时，团队仍然不知道怎么选。

## 组件与设计模式

组件解决单点问题，例如按钮、输入框、标签、弹窗。

设计模式解决一组任务，例如筛选排序、批量操作、审批流、消息通知、权限配置。

做设计系统时不要只沉淀组件，还要沉淀：

- 适用场景
- 不适用场景
- 正确/错误用法
- 状态和边界
- 与业务规则的关系
- 对应的前端实现方式

## Design Token

Design Token 是设计参数变量，用统一命名储存颜色、字号、间距、圆角、阴影、行高等基础样式。

Token 的价值：

- 统一设计和开发语言。
- 降低多端样式维护成本。
- 支持主题和品牌更新。
- 让设计规范真正进入工程系统。

常见 Token 类型：

- color
- font
- font-size
- line-height
- spacing
- radius
- border
- shadow
- opacity

命名建议：

- 不只按颜色命名，也要按语义命名。
- 例如 `color-brand-600` 是基础色阶，`color-text-primary` 是语义用途。
- 状态色要包含 success、warning、danger、info、disabled。

## 组件库放进作品集

组件库可以加分，但不要只放组件截图。

更有价值的表达：

- 为什么要做组件库。
- 解决了什么一致性或协同问题。
- 你负责了哪些模块。
- 如何定义 Token 和状态。
- 如何和开发对齐。
- 组件复用后带来什么效率或质量变化。

如果只是展示一堆按钮、输入框、卡片，面试官只能看到整理能力，看不到系统思考。

## 色彩系统

色彩系统要解决：

- 品牌识别
- 信息层级
- 状态反馈
- 可访问性
- 多场景扩展

不要只做一组好看的渐变色。至少要有：

- 基础品牌色阶
- 文本色
- 背景色
- 边框色
- 功能状态色
- 禁用和弱化色
- 明暗主题适配策略

## 视觉改进优先级

如果时间有限，先改：

1. 信息层级。
2. 字体和字号体系。
3. 间距系统。
4. 对齐和栅格。
5. 颜色语义。
6. 组件状态。

不要一上来加样机、阴影、渐变和装饰元素。

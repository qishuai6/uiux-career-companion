# 设计策略、问题定义与体验度量

## 核心判断

设计策略的价值是把“我要做什么”变成“为什么做、优先做什么、如何证明有效”。

设计师要能把业务目标、用户目标和设计目标区分开。

- 业务目标：增长、转化、留存、收入、成本、效率。
- 用户目标：完成任务、降低理解成本、获得信任、减少风险。
- 设计目标：信息触达、路径缩短、错误减少、一致性提升、满意度提升。

## 问题定义

不要直接接受表层需求。

例如用户说“按钮要放大”，真实问题可能不是按钮尺寸，而是用户想更快完成发送任务。

问题定义模板：

```text
【用户】在【场景】下，因为【障碍】，无法顺利完成【目标】，这会导致【业务/体验影响】。
```

继续追问：

- 用户是谁？
- 在什么场景下出现？
- 现在怎么解决？
- 为什么现有方案不够好？
- 这个问题影响多少人、多高频、多紧急？
- 解决后会出现什么可观察变化？

## POV

POV 用来从用户视角重定义问题。

基本结构：

```text
【用户】需要【需求】，因为【深层原因】。
```

重点不是满足用户提出的字面要求，而是找到背后的目标。

## HMW

HMW 用来把问题转成可发散的机会点。

常见角度：

- 积极：如何让用户更愿意完成任务？
- 否定：如何减少用户放弃的原因？
- 转移：是否能让系统或其他角色分担任务？
- 分解：能否把复杂问题拆成几个步骤？
- 脑洞：如果不受当前限制，可以怎么做？

HMW 之后必须做优先级评估，不要把发散当结论。

## GSM

GSM 是从目标到指标的拆解方法。

- Goal：目标是什么。
- Signal：目标实现后，会出现什么用户行为或态度变化。
- Metric：用什么指标衡量这些信号。

使用方式：

1. 先写业务目标、用户目标、设计目标。
2. 找出目标达成后的可观察信号。
3. 再选择指标。

避免一上来穷举指标。指标必须服务目标。

## HEART

HEART 适合从用户体验维度选择指标：

- Happiness：满意度、主观感受。
- Engagement：参与度、使用深度。
- Adoption：新功能/新产品采用。
- Retention：留存、回访。
- Task Success：任务成功率、效率、错误率。

不是每个项目都要用完整 HEART。选择和目标最相关的维度即可。

## 体验度量

体验度量分两类：

主观评价：

- 满意度
- NPS
- 问卷评分
- 访谈反馈
- 应用商店评价

客观度量：

- 任务成功率
- 完成时长
- 点击率
- 转化率
- 跳出率
- 错误率
- 投诉量
- 工单量

## 设计验证表达

作品集和面试中可以这样说：

```text
这个项目的业务目标是【目标】。我把它拆成设计目标【目标】，对应的用户信号是【行为/态度变化】，因此选择【指标】作为验证方式。
```

如果没有上线数据：

```text
因为项目没有完整上线闭环，我没有直接业务数据，所以用了【专家走查/可用性测试/关键用户反馈/任务路径对比】验证方向。这个验证不足也是我复盘中会补的部分。
```

## 竞品分析

竞品分析不是截图对比。

先明确：

- 分析目的是什么？
- 对比对象为什么是它？
- 对比维度服务哪个设计决策？
- 得出的机会点能不能落到当前项目？

可比维度：

- 目标用户
- 核心任务路径
- 信息结构
- 转化策略
- 关键交互模式
- 商业模式
- 体验指标

不要把竞品分析写成“他们有什么，我们有什么”。
